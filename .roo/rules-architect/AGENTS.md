# Project Architecture Rules (Non-Obvious Only)

- Система разделена на два основных сервиса: API (REST) и DLNA сервер (UPnP). Они работают независимо, но координируются через MainStreamManager.
- Зависимости управляются через Dependency Injection (библиотека `injector`). Это позволяет легко заменять реализации для тестирования и расширения.
- Обнаружение устройств: Яндекс.Станции через mDNS (библиотека `zeroconf`), DLNA рендереры через UPnP (библиотека `upnpclient`). Устройства кэшируются в DeviceManager.
- Поток стриминга:
  1. Яндекс.Станция отправляет статус трека через WebSocket.
  2. MainStreamManager получает трек, определяет его тип (радио или Яндекс.Музыка).
  3. Для Яндекс.Музыки запрашивается URL трека через YandexMusicAPI (если токен есть).
  4. URL отправляется на DLNA сервер через HTTP POST `/set_stream`.
  5. DLNA сервер загружает аудио и транслирует на DLNA рендерер через UPnP AV Transport.
- Архитектура addon Home Assistant:
  - Использует s6-overlay v3 как init систему.
  - Требует `init: false` в config.yaml, чтобы избежать конфликта PID 1.
  - Сервисные скрипты должны быть исполняемыми и расположены в `services.d/` (удалены в последней версии, заменены на run.sh).
  - Добавок поддерживает только архитектуры `aarch64` и `amd64`.
- Сетевой режим: В Docker используется `host` network mode, чтобы сервисы могли общаться с устройствами в локальной сети через multicast.
- Конфигурация: Все настройки вынесены в класс `Settings` (pydantic-settings), загружаются из `.env` с префиксом `APP_`.
- Тестирование: Используется `pytest-asyncio` для асинхронных тестов. Мокируются внешние зависимости (Yandex Station, DLNA устройство).
- Критические точки отказа:
  - Отсутствие токена Яндекс.Музыки отключает стриминг треков (радио продолжит работать).
  - Отсутствие DLNA рендерера в сети приведёт к ошибкам при попытке транслирования.
  - Сбои в WebSocket соединении с Яндекс.Станцией могут прервать стриминг (автоматический перезапуск через `_wrap_streaming`).
